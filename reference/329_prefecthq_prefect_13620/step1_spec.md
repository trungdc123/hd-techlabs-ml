# Prompt Category

Refactor, Performance.

# Repo Definition

Prefect is an open-source workflow orchestration framework written in Python. The area relevant to this task is the deployment CLI (`src/prefect/cli/deploy.py`), the flow loading utilities (`src/prefect/flows.py`), the callable introspection utilities (`src/prefect/utilities/callables.py`), and the import tooling (`src/prefect/utilities/importtools.py`). During a `prefect deploy`, the CLI currently imports and fully executes the user's flow module in order to read the flow's name, description, and parameter schema. This means every top-level side effect in the user's script runs at deploy time, and any missing runtime dependency causes the deploy to fail even though the deploy process never actually invokes the flow.

# PR Definition

The PR replaces the runtime import of the user's flow module during deployment with static, AST-based extraction. Where the deploy CLI previously called `load_flow_from_entrypoint` (which executes the entire module to obtain a `Flow` object), it now calls two lighter-weight alternatives. First, `load_flow_argument_from_entrypoint` in `flows.py` parses the source file into an AST, locates the decorated function by name, and reads keyword arguments like `name` and `description` directly off the `@flow(...)` decorator node, falling back to the function name (with underscores replaced by hyphens) when no explicit name is given. Second, `parameter_schema_from_entrypoint` in `callables.py` uses `safe_load_namespace` (added in `importtools.py`) to resolve imports and class definitions with best-effort error suppression, then builds an `inspect.Signature` from the AST function definition and generates the OpenAPI parameter schema without ever calling the function. The deploy CLI also drops the mutual-exclusion check between `flow_name` and `entrypoint` because the entrypoint is now always used as the authoritative source. The net effect is that deploying a flow no longer executes the user's module, does not require all runtime dependencies to be installed in the deploy environment, and is significantly faster for modules with expensive top-level work.

# Edge Cases

The AST-based decorator reader only recognizes a bare `flow` identifier on the decorator call, so if the user writes `@prefect.flow(name="x")` or aliases the decorator under a different name the function will not find the keyword argument and will fall back to the function name. The fallback itself only applies to the `name` argument — requesting `description` when no decorator keyword is present returns `None`, which callers must handle. The entrypoint parser splits on the last colon, which correctly handles Windows drive-letter paths like `C:\path\to\file.py:my_flow`, but a module-style entrypoint containing a colon in a dotted path would be misrouted into the file-path branch.

The `safe_load_namespace` utility swallows `ImportError` for missing packages and generic `Exception` for class and function compilation, which means annotation evaluation degrades silently. If a type annotation or default value references something that cannot be imported or evaluated, the signature builder drops the annotation or sets the default to `None` rather than raising, so the resulting parameter schema may be less precise than the one produced by the old runtime path. This is an intentional trade-off, but reviewers should watch for scenarios where the schema difference causes the Prefect UI or API to reject a deployment that previously worked.

The `_generate_signature_from_source` helper uses `ast.walk`, which does not respect scope boundaries, so a nested function definition with the same name as the target would match first depending on tree-walk order. It also only handles `ast.FunctionDef`, not `ast.AsyncFunctionDef`, meaning an `async def` flow will not be found and will raise a `ValueError`.

# Acceptance Criteria

A correct implementation should let a user run `prefect deploy` against a flow whose module has uninstalled third-party imports at the top level, and the deploy should succeed as long as the flow function's own signature and decorator arguments are expressible through the available namespace. The flow name must match whatever is passed to `@flow(name=...)`, falling back to the function name with underscores replaced by hyphens when no explicit name is given. The description should come from `@flow(description=...)` and default to `None`. The parameter schema must be functionally equivalent to what `parameter_schema(fn)` would produce when the function can actually be imported — covering basic types, `datetime`, `pendulum`, `Enum` subclasses, Pydantic models, `SecretStr`, generic typing constructs, default values, and docstring-based parameter descriptions.

A reviewer would know the implementation is correct if the full `TestLoadFlowArgumentFromEntrypoint` and `TestEntrypointToSchema` suites pass, including the case where a missing import (`bipitty_boopity`) does not cause a failure. The deploy CLI path should no longer reference `load_flow_from_entrypoint` or `parameter_schema` directly, and should not call `run_sync_in_worker_thread` for flow loading. An implementation would be incomplete if it still executes the user module during deploy, if it does not handle the `@flow` decorator without arguments (bare `@flow`), or if it fails to propagate docstring descriptions into the schema. It would be incorrect if it silently drops required parameters, mangles default values for evaluable expressions, or raises on a missing third-party import that the flow's own signature does not depend on.

# Initial Prompt

The Prefect deployment CLI currently imports and executes the user's entire flow module just to read the flow name, description, and parameter schema. This is slow, triggers top-level side effects, and fails when runtime-only dependencies are not installed in the deploy environment. Refactor the deploy path so that none of these values require executing the module.

The deploy process needs a way to extract the flow name and description from the `@flow(...)` decorator without importing the module. This should work by parsing the source file as an AST, locating the target function definition, and reading keyword arguments directly off the decorator node. When the decorator is bare (`@flow` without a call) or the requested keyword is absent, the name should fall back to the function name with underscores replaced by hyphens, and the description should default to `None`. Both `path:func` and `module.path.func` entrypoint formats need to be supported, splitting on the last colon or dot respectively.

The deploy process also needs a way to generate the parameter schema from an entrypoint without instantiating the flow. This requires building a best-effort namespace from the source code — attempting each import statement and class/function definition with errors swallowed so that missing third-party packages do not cause failures. Using that namespace, construct an `inspect.Signature` from the AST function definition by evaluating annotations and defaults (falling back gracefully when evaluation fails), extract the docstring from the AST, and generate the OpenAPI parameter schema. The existing `parameter_schema` function that works on live callables should share the same schema-generation logic so behavior stays consistent.

With these capabilities in place, update the deploy CLI to use the static alternatives instead of `load_flow_from_entrypoint` and `parameter_schema(flow)`. The mutual-exclusion check between `flow_name` and `entrypoint` can be dropped since the entrypoint is now always the authoritative source.

Add tests covering decorator argument extraction (explicit name, implicit name fallback, explicit description, missing description, missing function), entrypoint-based schema generation (no arguments, required and optional arguments, annotated types, datetime and pendulum types, enums, generics, user-defined classes, Pydantic models, `SecretStr`, docstring extraction, and a missing third-party import that should not cause failure), and the safe namespace loading utility (successful imports, failed imports, and failed class definitions).
